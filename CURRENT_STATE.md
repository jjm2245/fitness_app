**Version-specific gotchas a fresh agent WILL trip on:**

- **Middleware → "proxy" rename.** Next 16 renamed the middleware convention.
  The file is **`src/proxy.ts`** exporting a function named `proxy` (not
  `middleware`). `AGENTS.md` warns: *"This is NOT the Next.js you know … Read the
  relevant guide in `node_modules/next/dist/docs/` before writing any code."*
  Take that literally.
- **Proxy runs on the Edge runtime → no `node:crypto`.** `src/lib/auth.ts` uses
  Web Crypto (`crypto.subtle` HMAC-SHA256) so the same helpers work in both the
  Edge proxy and the Node-runtime API routes. Don't import `node:crypto` into
  anything the proxy touches.
- **Route handler params are async.** `{ params }: { params: Promise<{ id: string }> }`
  — you must `await params`. All dynamic routes here do this.
- **`useSearchParams` needs a Suspense boundary** or the build errors on the
  statically-rendered route. `src/app/login/page.tsx` wraps its form in
  `<Suspense>` for exactly this reason.
- **ESLint flags `setState` called synchronously inside a `useEffect` body**
  (React-compiler rule `set-state-in-effect`). Established fix in this repo: put
  the async work in an IIFE inside the effect and only `setState` after an
  `await` (see effects in `log/[id]/page.tsx`, `exercises/page.tsx`). Also:
  never name a plain function `useXxx` — the hooks linter treats it as a hook
  (bit us once; renamed `useLibraryName` → `adoptLibraryName`).

## 2. Run it locally (assume nothing is installed)

Node 20 + Postgres 16 live in a **conda env** (this machine had no
Node/Postgres/Docker/Homebrew). Postgres is a plain local data dir, **not** a
system service, on **port 5433** via a Unix socket in `/tmp`.

```bash
conda activate fitness-app                       # env name is "fitness-app"
# start / stop local Postgres (run from repo root):
pg_ctl -D .pgdata -l .pgdata/logfile -o "-p 5433 -k /tmp" start
pg_ctl -D .pgdata stop

npm install
cp .env.example .env      # then fill values (see env vars below)
npm run db:migrate        # apply all drizzle/*.sql
npm run db:seed           # curated exercise graph + starter program/blocks (only if none exist)
npm run db:seed:library   # ingest free-exercise-db (idempotent) + apply merges
npm run dev               # http://localhost:3000, gated by APP_PASSCODE
```

## 6. Offline / sync architecture (`src/lib/sessionStore.ts`) — most bug-prone, read carefully

**IndexedDB `fitness-app-session`, version 4.** Object stores:

| Store | Key | Notes |
|---|---|---|
| `sessions` | `id` (client session id) | `LocalSession` (date, origin=aggregated name, finishedAt, finishSynced) |
| `occurrences` | `instanceId` | ordered performed list; index `by-session`; each has `synced` flag |
| `sets` | auto `localId` | indexes `by-session`, `by-instance`; carries `syncState` + `serverId` |
| `cardio` | auto `localId` | same shape as sets |
| `completed` | `instanceId` | per-occurrence "done" flag |

**Sync state machine** (per set/cardio row): `pending_create` → `synced` →
(`pending_update` | `pending_delete`). Occurrences use a boolean `synced`.

**`sync()` (exported) is serialized** — it chains each drain after the previous
(`syncChain`) so two rapid callers can't both read the same pending rows and
double-POST (set-logs is a plain insert → that duplicated logged sets; this was a
real data-integrity fix). Internally `runSync()` does, **in this order**:
1. **Occurrences first** (so set-logs can resolve `session_exercise_id`): one
   `POST /api/session-exercises` per session with the full ordered list; marks
   all `synced`.
2. sets (create/update/delete), 3. cardio, 4. finish stamps, 5. **queued session
   deletes** (localStorage `fitness-app:pending-session-deletes`, drained via
   `DELETE /api/sessions/[id]`, idempotent).

**Failure classification** — every request goes through `send()`, which throws a
typed error: `auth` (HTTP 401), `network` (fetch threw), `server` (other non-ok;
404 allowed on deletes). `runSync` returns `{created,updated,deleted,finished,
failed, authError, networkError, serverError}`. **On the first `auth` error it
aborts the whole drain** (every later request would 401 too) and the UI shows a
re-login link (`/login?next=…`); it auto-re-drains on `online` + tab
`visibilitychange`.

**Hydration** — opening a session not in the local store fetches
`GET /api/sessions/[id]` and `hydrateFromServer` rebuilds occurrences (in order)
+ sets/cardio as `synced` with server ids, re-linking each set to its occurrence
by `session_exercise_id` (falling back to the first occurrence of that exercise;
legacy sessions with no `session_exercises` get one synthesized occurrence per
distinct logged exercise). Never clobbers a session that already has local state.

**Session delete is offline-safe**: `deleteSession(id)` removes local rows now +
queues the server delete in **localStorage** (no IndexedDB version bump), drained
on next sync.

**`pendingCount(sessionId?)`** counts unsynced sets + cardio + unsynced
occurrences + unsynced finishes + queued deletes.

Footguns here:
- The DB is on **v4**; opening it wipes *unsynced* local data on a version bump
  (finished sessions are safe on the server). Any future store change that needs
  a bump is destructive for in-flight local data — be deliberate.
- `_resetDbForTests()` is exported **for tests only** (deletes the whole IDB).
- `localStorage` is used for the delete queue → tests stub it.

## 11. Traps (things that waste time or cause damage)

- **Don't `next build` while `next dev` is running** — it clobbers `.next` and
  the dev server starts 500ing. Stop dev first.
- **Destructive commands:** `DELETE FROM <table>` without a predicate; the
  auto-mode classifier blocks unqualified deletes for good reason. `TRUNCATE …
  CASCADE`. Re-running `db:seed:library` re-derives library rows (idempotent, but
  it *does* delete+recreate `exercise_muscles` for each row).
- **IndexedDB version bumps are destructive to unsynced local data.** Bump v4→v5
  only when you mean it, and document it.
- **`sync()` must stay serialized.** If you refactor it, keep the `syncChain`
  (or an equivalent lock) or you'll reintroduce duplicate logged sets.
- **Occurrence ordering + set linking:** occurrences must sync *before* sets, or
  `set_logs.session_exercise_id` won't resolve. Preserve that order in `runSync`.
- **`workout_logs.date` is `NOT NULL`** but sessions are keyed by
  `client_session_id`; don't reintroduce date-as-identity assumptions.
- **The proxy matcher** excludes `login`, `api/auth/login`, static assets. If you
  add a public route, update the matcher in `src/proxy.ts`.
- **conda PATH:** a fresh shell won't have `pg_ctl`/`psql`/the right `node`;
  activate the env or prepend the env `bin`.
- **Secrets:** `.env` holds real values (gitignored). The Neon password appeared
  in prior session transcripts and a rotation is outstanding — don't paste
  connection strings into committed files or logs.
