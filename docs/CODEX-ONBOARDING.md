## 1. What this is

A **private, single-user fitness app** (mobile-first PWA, personal use — not a product for other users). Its core is an *agent* that:
- designs and continuously re-optimizes a lifting program from evidence-based principles,
- adapts in real time to changing constraints (travel, limited equipment, fewer days, a tweaky joint),
- and over later phases folds in recovery, bodyweight, nutrition, and progress photos to steer the user's whole physique trajectory.

**The user:** male, born 1999, ~170 lb, 5'11", <1 year training, goal = hypertrophy/recomposition (aesthetics-first, explicitly *not* strength/PRs). Trains 6 days/week PPL, ~1 hr lifting + 15–30 min abs/cardio, primarily at Planet Fitness (plus barbell access elsewhere). Minor non-hindering back discomfort. Sedentary desk job. Owns an Oura ring.

**The wedge (why build this at all):** the market is saturated with broad-but-shallow all-in-one AI fitness apps that wrap a generic LLM around a logger. This project's differentiator is a **rigorous deterministic programming engine** with a reasoning layer on top — depth where the others are thin.

### Why Milestone 3 (the agent) is deferred — do not "helpfully" start it
The plan is to **log real sessions for a couple of weeks first**. Building AI coaching on top of progression thresholds that haven't been validated against real training would mean reasoning over an empty/fake history and cementing untested defaults. **Do not add any LLM or vision code unless explicitly asked.** This is the single most important scope rule in the project.

### Key later-phase design decisions already made (see spec for detail)
- **Nutrition:** tiered source-of-truth — Tier 1 authoritative (USDA Branded + Open Food Facts for packaged; Nutritionix for chain restaurants), Tier 2 USDA generic (portion-dependent), Tier 3 flagged low-confidence (independent restaurants/mixed dishes). Food photos return a **range + confidence**, never a hard number. Maintenance calories derived **empirically** from 7-day intake vs. bodyweight trend, not a formula.
- **Oura:** personal access token, scheduled server-side pull. A **recovery *modifier*** on volume/deloads — not a hypertrophy dial.
- **Dashboard:** transparent sub-scores; a flat scale during recomp is **success**, not failure, and the UI must say so.
- **Form analysis:** assistive check, never a verdict or injury clearance; on-device pose estimation for objective geometry, LLM only for gross errors.

## 3. The architectural philosophy (load-bearing — do not violate)

**The agent is three layers, not a chatbot.**

```
Reasoning layer (Claude API, server-side)  ← judgment, language, perception ONLY
        ↓ tool calls
Deterministic core (tested backend code)   ← ALL math and training rules
        ↓
Data layer                                 ← exercise graph, logs, program, metrics
```

**Why:** if the LLM re-derives "how many sets per week" every message, it drifts and hallucinates. Numbers and rules stay deterministic and testable; the model handles language, judgment, and perception.

### The rules that follow from this
1. **`src/core/*` must stay fully general and data-driven.** It has been audited: **zero** exercise IDs, day names, muscle names, or rep-range literals outside test fixtures. Every function takes plain tagged data. **Self-check this every session.** The user's routine lives in *data* (seed + DB), never in engine logic.
2. **A program is user-owned editable data, not app structure.** The user's PPL is "a program they created," not the app's identity. Programs/blocks are *optional scaffolding* — logging is the primitive. Ad-hoc exercises attach to the **session**, never to a phantom program.
3. **Single-user.** No multi-tenant logic, no per-user auth. One passcode.
4. **Offline-first is non-negotiable.** Gyms are signal dead zones. Every flow (logging, finishing, editing, adding a machine) must work offline and sync later. The UI reads from the local store, never a network round-trip.
5. **Loads are context-bound.** Cable/machine/Smith loads are **not** comparable across machines (pulley ratios: 1:1 vs 2:1 halves the real resistance; the PF Smith bar ≈15–20 lb, not 45). Track progression *per machine*; **re-baseline on machine change** rather than flagging a false stall. Free weights/bodyweight are the **portable anchors**. Effort + volume are the universal currency.
6. **Progression = volume-load (load × reps)**, never estimated-1RM (unreliable in the 12–20 rep hypertrophy zone and on cables/Smith). Rising reps at a fixed load **is progress**, not a stall.
7. **Structural exclusion over remembered filters.** E.g. `cardio_logs` is a separate table so the core (which reads only `set_logs`) is *physically* incapable of counting cardio toward volume. Untagged customs are excluded because they have no movement pattern — not because a filter remembers to skip them. Prefer designs where the guarantee is structural.

## 4. Domain context you cannot infer from the code

- **Movement pattern** is the tag the substitution engine runs on (horizontal_push, vertical_pull, hinge, squat, etc.). It's what makes a pull-up and a lat pulldown interchangeable. **The open library dataset (`free-exercise-db`, Unlicense) has no movement-pattern field** — that's why library exercises can't substitute until tagged. The ~33 hand-curated exercises have patterns; the other ~857 don't.
- **"Curated vs library" is not a real distinction** — it's **fully-tagged vs partially-tagged**. Any exercise graduates to first-class the moment it gets a movement pattern. Exercises get tagged **on add**, only the ones actually used.
- **Canonical name = the library's standard name.** A paired exercise is **one entry** that *displays the canonical name* and *keeps the curated tags*. Prefer the library's cleaner name; keep the user's richer tags.
- **Some canonical names are knowingly imperfect** (a "watch-list" in `DECISIONS.md`) — e.g. the user does machine chest press *seated* not flat; uses a hip-height not low cable for lateral raises; uses a close-grip double-D handle on seated rows. **Do not "fix" these** — they're accepted tradeoffs.
- **Planet Fitness reality:** no loadable barbell or squat rack at standard locations — Smith machine instead; dumbbells to ~75 lb; fixed bars to ~60 lb; cables; extensive selectorized machines. This is *fine* for hypertrophy. The user also has barbell access elsewhere.
- **Back safety:** `affected_structures: ["lumbar_spine"]` tags exist on hinges/back extensions/loaded rotation. Substitution must exclude by **affected structure**, not just movement pattern. The app must never imply medical clearance.
- **Effort tag replaced RIR:** the UI shows *more in me / near failure / to failure*; an adapter maps it to a normalized number the core consumes. The user defines failure as "cannot complete another rep." Don't reintroduce raw RIR as primary.

## 5. How we work (process — follow this)

This process has repeatedly caught real bugs. Honor it.

1. **Read the spec + `DECISIONS.md` + `CURRENT_STATE.md` first.** The spec is authoritative; it encodes reasoning you won't reconstruct on your own. **If your instinct contradicts the spec, the spec probably wins — raise it, don't silently deviate.**
2. **Propose a plan and wait for explicit go-ahead** before building. Include schema approach and sequence. For big batches, propose **checkpoints** and pause at each.
3. **Stay in scope.** Build what's asked, nothing from later phases. No LLM/vision unless explicitly requested.
4. **Migrations run local-first. Production is held until the user reviews locally.** Never migrate prod unprompted.
5. **Never mass-delete data without explicit sign-off.** (A guardrail correctly blocked a `TRUNCATE` earlier — that's the desired behavior.)
6. **Keep IDs stable.** When splitting/merging/renaming exercises, never orphan logged history. Removals must warn and preserve past logs.
7. **Verify in the running app, not just at the data layer.** Several bugs were "correct in the DB, broken on screen." If the symptom was visual, the proof must be visual.
8. **Write tests** alongside core logic; don't regress the suite. Run typecheck/lint/`next build`.
9. **Update `DECISIONS.md`** with every decision and deviation. It is the shared memory between agents and the user.
10. **Commit at clean checkpoints**, one logical chunk per commit, so a bad change can be rolled back precisely.
11. **The user validates on a real phone.** Offline behavior, install-to-home-screen, and gym usability can only be proven on the device — flag explicitly what needs their hands.

### Things that are *always* off-limits without explicit approval
- Adding any LLM/vision call · touching `src/core/*` generality · migrating prod · mass-deleting rows · introducing multi-user/auth complexity · starting the design/gamification overhaul · building later-phase features (Oura, nutrition, photos, dashboard, form analysis).

## 6. Build history — the short version

Sessions have run roughly: foundations + deterministic core → logging UX v1 → **audit** (confirmed core is general; found two leaks in the seed/API layer, since retired) → program editor + logging redesign (routine demoted from "app structure" to editable data) → deploy to phone (managed Postgres, hosted, HTTPS, installable) → interaction model (confirm set / confirm exercise / **finish with summary**) → reusable blocks + cardio logging + library integration → **sessions list** as home base → **session model v2** (ordered, incremental, occurrence-based) → **real-world logging on the phone**, which drove: a run of sync-integrity fixes (false "not synced", occurrence-list `occurrencesDirty` model, history-safe prune + directional heals, additive-migration data-loss guard), the **rest-edge model** (restBefore + set-level timer, honest-unknown), **drop sets** + **unilateral sides**, the **Equipment model** (type vs unit, per-unit offsets, opaque lane keys — core still equipment-agnostic), **stable/editable session date-time** (local-time everywhere), and **server-persisted completion + equipment** so a finished session survives a PWA reinstall.

### Bugs found that shaped the design (understand these — they're the failure modes here)
- **Silent sync loss.** An expired cookie caused `/api/*` to *redirect* to `/login`, which returned **200 HTML** — so `res.ok` was true, `res.json()` threw, the error was swallowed, and data sat "not synced" forever. Fix: API routes return **401 JSON**, failures are **classified** (auth/network/server) and **surfaced**, and the outbox re-drains after re-login. **Lesson: a failure disguised as success is the worst bug class here — never swallow sync errors.**
- **Double-insert on rapid logging.** Two overlapping `sync()` calls read the same pending rows and inserted twice — silently inflating volume. Fix: serialized drains. **Lesson: concurrency in the offline outbox is dangerous; test it.**
- **Destructive seeding.** The seeder wiped and recreated `program_exercises` on every run, which would have erased user edits. Fix: seed only when no program exists.
- **Date-keyed sessions.** Sessions keyed by calendar date would collide for two sessions in one day. Fix: client-generated session IDs.
- **Half-applied pairing.** Merging seed→library left duplicate rows visible with no signal which to pick. The fix is **one merged entry** (canonical name + curated tags).

## 7. Coordinating two coding agents (you + Claude Code)

Real risk: two agents editing the same repo produce conflicting decisions and merge conflicts.

- **Work on separate branches, or separate scoped areas — not the same files simultaneously.** The user should tell each agent what it owns for a given session.
- **`DECISIONS.md` is the shared contract.** Read it at session start; append to it at session end. If it conflicts with your plan, raise it.
- **`CURRENT_STATE.md` may be stale** if the other agent shipped since it was generated — re-verify against the repo before relying on details.
- **Don't rewrite the other agent's decisions** because you'd have done it differently. If something looks wrong, flag it to the user with reasoning; don't unilaterally "fix" a documented choice.
- **Pull/rebase before starting**, and commit in small logical chunks so conflicts stay tractable.
