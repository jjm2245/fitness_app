# SPEC-DRIFT.md — where the built system diverges from the spec's intent

**Purpose.** [`fitness-agent-spec.md`](fitness-agent-spec.md) (v0.5) is the
**authoritative statement of intent** and is human-owned. It predates a lot of the
built system. This file reports, section by section, where the code has diverged
from or outrun the spec — **what the spec says vs. what is true** — so the owner
can fold the real decisions into v0.6.

This is a **signal, not a bug list.** The spec is not auto-synced to the code on
purpose: if it silently tracked reality it would stop being a check on the code
and drift would stop being information. Each item below is a "did we mean this?"
prompt for the spec owner, not something an agent should "fix" by editing the spec.

Generated by Claude Code from the built system; no code changes implied. Ordered
by spec section.

---

## §5 — Agent design (deterministic vs. LLM)

- **Still accurate in principle** (three layers; core owns the math; LLM deferred).
- **Drift — substitution's division of labor.** §5/§8 read as if the hand-curated
  candidate list is the substitution source. In reality the **deterministic engine
  computes candidates** (movement pattern + muscle overlap + available equipment +
  injury structures); the hand-curated `exercise_substitutions` list is **inert**
  (`candidate_exercise_id` never resolved). The intended end state (consistent with
  the layering) is: engine *filters*, the future agent *picks among* candidates
  using judgment — and the curated **notes** ("leg press if back flares", Pallof as
  the lumbar-safe rotation) are LLM-material, not core-material. Worth stating
  explicitly in v0.6.

## §6 — Data model (the biggest drift)

The built schema has moved well beyond the spec's block. Concretely:

- **`Machine` → `Equipment`, and a type-vs-instance model the spec doesn't have.**
  Spec: `Machine(id, gym, brand, model, pulley_ratio, counterweight_lb, cam_profile,
  notes)`. Built: an `equipment` table with a **surrogate id** (legacy rows keep
  label-as-id; new = uuid) + a display `label`, plus an **equipment *type*** concept
  (bodyweight | dumbbell | kettlebell | fixed/​olympic/EZ barbell | cable |
  selectorized | smith | plate_loaded) and a **built-in offset** (`builtInWeight`,
  additive — e.g. Olympic bar 45). "Instance matters?" is modelled as the inverse of
  `portable`. None of this taxonomy is in the spec.
  - **Two different offset concepts now exist:** the spec's `counterweight_lb`
    (Smith, *reduces* effective load, stays descriptive) vs. the new `builtInWeight`
    (bar/carriage, *adds* to effective load, folded into the number). v0.6 should
    name both.
  - **`pulley_ratio` is captured but deliberately kept OUT of the load math** — a
    multiplicative ratio cancels within a lane (loads are lane-scoped per unit), so
    folding it in would add false precision. The spec lists it as a field but doesn't
    state this decision.
- **RIR → effort tag (conceptual drift the owner already flagged).** Spec: `SetLog.rir`,
  `ProgramExercise.rir_target`, progression "≤ target RIR". Built: the **primary
  effort signal is a tag** — `more_in_me | near_failure | to_failure` — with `rir`
  kept as an *optional* exact number; the progression adapter reads the tag (derives
  an RIR when needed). Rationale (in DECISIONS): a 3-point tag is more honest to log
  mid-set than a precise RIR you don't actually know.
- **No occurrence / session-v2 model in the spec.** Spec: `WorkoutLog(date,
  program_day, entries[])` + `SetLog(exercise_id, …)`. Built: a session is **a thing
  you start**, not a calendar day — `workout_logs` keyed by `client_session_id`
  (unique, client-owned), with an ordered list of **occurrences** in
  `session_exercises` (`client_instance_id`, `order_index`, repeats allowed — the
  same exercise can appear multiple times), and sets link to their occurrence via
  `session_exercise_id`. Plus `first_finished_at` + `first_finished_source` (stable
  display date + user-editable time) and `completed` (the per-occurrence "done"
  check, server-persisted).
- **No rest model in the spec.** Built: `rest_seconds` + `rest_source`
  (`timed | derived | user`), stored as **restBefore** on each set (N sets ⇒ N−1
  rests; the boundary between exercises is not a rest), honest-unknown (null, never
  fabricated).
- **No drop-set / unilateral-side / timestamp fields in the spec.** Built:
  `drop_set_group` (nested drop sets), `side` (`left | right | both`, unilateral
  only), `logged_at` (client-stamped instant, drives rest derivation).
- **Set load semantics.** Spec `SetLog.load` is the raw number. Built: `load` is the
  **effective total** = `loadEntered` + `builtinOffset`, with the components stored
  and the math shown transparently ("100 + 25 = 125").

## §7 / §7a — Programming engine & program management

- **Mostly accurate**: volume-load progression, fractional set-counting, stall
  detection, program-as-editable-data all built as specified.
- **Drift — `defaultLoadIncrement` keys off `load_type`.** This is "the last taxonomy
  in the core." With the Equipment model, increment is really a property of the
  *unit*. Clean end state (deferred, recorded in DECISIONS): the adapter passes a
  per-unit increment so the core stops keying on `load_type` at all.
- **Program structure elaborated:** spec `Program(split_type, days[])`; built
  `programs` + `program_days` + `program_exercises`, plus a hidden container program
  holding reusable **blocks**. Targets (`target_sets`/`rep_range`/`rir_target`) live
  on `program_exercises`; note the `rir_target` naming persists though the UI is
  effort-tag-first.

## §8 / §8a — Adaptation / substitution & the PF equipment reality

- The PF equipment reality §8a describes narratively is now **modelled
  structurally** (equipment types + per-unit offsets + opaque **lane keys**). The
  key built abstraction the spec lacks: the core is handed an **opaque lane key**
  (`laneKey(type, unitId)` → a unit id, or `"<type>:unspecified"`, or `null` for the
  portable lane) and groups/re-baselines by it, **never learning what a Smith is**.
  This is how "loads are context-bound" is enforced without equipment concepts
  leaking into `src/core`.

## §9 — Machine & load portability

- **Matches and extends.** Per-lane tracking, re-baseline-on-change, and
  effort+volume as the universal currency are all built. Extensions not in the spec:
  the lane-key mechanism above; **recalibrate-with-continuity** ("you were at 140 on
  the other unit", never "no data"); and the pulley-ratio-excluded-from-math decision
  (see §6). "Unspecified unit of a type" gets **its own lane** (not the portable
  lane) — a distinction the spec doesn't draw.

## §13 — Tech stack

- Accurate (Next.js/Postgres/Drizzle/IndexedDB/Vercel/Neon). One addition worth
  recording: the **additive-IDB-migration** discipline + data-loss guard, and the
  **client-owned identity** pattern (session ids, occurrence instance ids, equipment
  ids) that makes offline-created rows map to exactly one server row.

## §15 — Build milestones & status

- **Stale**, but by design this is now delegated: live status lives in
  [`CURRENT_STATE.md`](CURRENT_STATE.md) (§9 gaps / in-flight). v0.6 should drop the
  per-milestone status from the spec and point at CURRENT_STATE, keeping only the
  milestone *definitions* (intent).

## Not drifted (confirmed still true)

Single-user; offline-first; core generality (audited, zero equipment/routine
literals); volume-load over estimated-1RM; structural exclusion (cardio in its own
table); the deferral of the agent layer, recovery, nutrition, body metrics, photos,
and form analysis. §1 (profile/goals), §10–§12a (later-phase designs), and §14
(privacy/safety) are untouched by the build and remain accurate intent.
