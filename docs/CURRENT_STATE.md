## 1. Stack & versions

- **Next.js 16** (App Router; middleware is `src/proxy.ts` — the "proxy" rename;
  Edge runtime uses Web Crypto not `node:crypto`; dynamic-route `params` are a
  `Promise` and must be awaited). **React 19.** **TypeScript 5.**
- **Postgres** via **Drizzle ORM** + `pg` driver. Schema in `src/db/schema.ts`,
  migrations in `drizzle/` (drizzle-kit), applied with `npm run db:migrate`.
- **IndexedDB** via `idb` — the offline store, **version 4**, in
  `src/lib/sessionStore.ts`. This is the load-bearing offline layer.
- **Vitest** (unit) with `fake-indexeddb`. No E2E/browser test runner — UI is
  verified by driving the local dev server + manual phone testing.
- **PWA:** `public/sw.js` (hand-written service worker) + `manifest.webmanifest`.
- **Prod:** Vercel (auto-deploys `main`) + Neon Postgres. Live at
  **https://fitness-app-self-pi.vercel.app** (health: `/api/health`).

## 2. Run it locally

Requires Node ≥20 and a local Postgres. Env in `.env` (never committed):
`DATABASE_URL`, `APP_PASSCODE`, `SESSION_SECRET`.

```bash
npm install
npm run db:migrate        # apply migrations to DATABASE_URL
npm run db:seed           # curated exercise graph (idempotent)
npm run db:seed:library   # free-exercise-db ingest (slow; idempotent)
npm run dev               # http://localhost:3000  (or: preview_start "fitness-app")
npm test                  # vitest
npm run build             # next build (must be clean)
npm run db:check          # fail if DB schema is behind the code's migrations
npm run docs:refresh      # regenerate the AUTOGEN block in this file
npm run docs:check        # fail if the AUTOGEN block is stale
```

`release` = `db:migrate && db:seed && db:seed:library`. Neon's **pooled**
endpoint chokes drizzle-kit migrations — use the **direct** endpoint (strip
`-pooler` from the host) for `db:migrate` against prod. **Shipping a schema change
to prod: follow [`RUNBOOK.md`](RUNBOOK.md)** (migrate-before-deploy order, the
endpoint split, before/after counts, the health/db:check gates).

## 4. Data model (current — migration count in the AUTOGEN block)

Server (Postgres) tables of interest — full list in the AUTOGEN block:

- **`exercises`** — the graph node. `source` (curated|library|custom),
  `movementPattern` (nullable — null ⇒ `untagged`, not substitutable),
  `canonicalName`/`libraryId` (library pairing), `affectedStructures[]` (e.g.
  `lumbar_spine`), `unilateral`, `description`, `loadType`, `portable`. Muscles in
  `exercise_muscles` (fractional emphasis). `exercise_substitutions` holds the
  **hand-curated** candidate list — **mostly inert** (see §9).
- **`equipment`** (renamed from `machines`, migration 0016) — **surrogate `id`**
  (legacy rows keep label-as-id; new = uuid) + `label` (display) + `equipmentType`
  + `builtInWeight` + `pulleyRatio`/`counterweightLb`/etc + free-text `notes`.
  `exercise_equipment` join. Never rename the id (history-safe); rename = edit
  `label`; merge duplicates = re-point references then delete (like exercise
  collapse).
- **`workout_logs`** — a session. `clientSessionId` (client-owned identity,
  unique), `date` (stable calendar day — the display/sort anchor), `finishedAt`
  (re-stamps on every finish — **not** a display anchor), **`firstFinishedAt`**
  (stamped once; the display/sort time anchor) + **`firstFinishedSource`**
  (`'user'` when set via the date/time editor).
- **`session_exercises`** — the ordered performed **occurrences** (session-model
  v2). `clientInstanceId` (client-owned), `orderIndex`, `source`, **`completed`**
  (the "done" check — synced so it survives a PWA wipe). The same exercise can
  occur multiple times (repeats).
- **`set_logs`** — one row per set. `sessionExerciseId` (occurrence link,
  `set null` on occurrence delete), `load` (the **effective total** = entered +
  built-in offset), `loadEntered` + `builtinOffset` (the components), `reps`,
  `effort` (the primary effort tag; `rir` optional exact), `setIndex`,
  `setType`, `equipmentId` + `equipmentType`, **`loggedAt`** (client-stamped —
  drives rest derivation), **`restSeconds`** + **`restSource`** (`timed|derived|
  user`; null = honest-unknown, never fabricated — stored as *restBefore*, so N
  sets ⇒ N−1 rests), **`dropSetGroup`** (drop-set link), **`side`** (`left|right|
  both` for unilateral).
- **`cardio_logs`** — separate shape (duration/incline/speed/…); structurally
  invisible to the core (core reads only `set_logs`).
- Later-phase tables exist but are unused: `body_metrics`, `recovery_metrics`,
  `nutrition_entries`, `progress_photos`, `form_checks`, `injury_flags`, `profile`.

Local (IndexedDB v4) stores: `sessions`, `occurrences` (keyed `instanceId`),
`sets`, `cardio`, `completed` (**legacy — completion now lives on the occurrence;
this store is a dead mirror, slated for removal**).

## 5. API surface

See the AUTOGEN routes table. Shape notes: occurrences sync via
`POST /api/session-exercises` (whole ordered list; prunes removed rows but
**refuses to delete an occurrence that still has logged sets** — the wrong-side-
wins guard, reporting `keptWithHistory`). `GET /api/sessions/[id]` returns the
full session (occurrences + sets + cardio) for hydration. `PATCH
/api/sessions/[id]` edits date/`firstFinishedAt` (source `user`). Set/cardio
create+PATCH carry all logging-depth fields.

## 6. Offline / sync architecture (`src/lib/sessionStore.ts`) — most bug-prone

The UI **always reads the local store**, never a network round-trip; `sync()`
reconciles in the background. `sync()` is **serialized** via `syncChain` (two
rapid calls must not double-POST — that duplicated logged sets historically).

**Drain order (matters):** occurrences → sets → cardio → **second occurrence
pass** → finish → date/time (meta) → queued deletes. Occurrences go first so
set-logs can link by `clientInstanceId`. The second occurrence pass exists
because a set-bearing occurrence the client removed is *kept* on pass 1 (guard),
its sets delete in the set loop, then pass 2 prunes the now-empty row — heals a
legit remove-with-sets in one sync.

**Dirty flags & pending signals** (all surfaced in the sessions-list "not synced
· <reason>" badge):
- `occurrencesDirty` (session-level) — the ordered list has an un-pushed change
  (add/remove/reorder). Dirtiness is a property of the *list*, so it survives
  removing the last occurrence. Cleared when the list POSTs cleanly.
- `occurrenceConflict` — the server kept occurrence(s) this device tried to drop
  because they carry logged sets the device doesn't have ⇒ **this device is
  behind**. Heal = **Pull from server** (`rehydrateLocalFromServer`), not
  Reconcile.
- finish arm: `finishedAt && !finishSynced` (reconciled from the server on the
  sessions page — a server-confirmed finish never shows false "not synced").
- `metaDirty` — un-pushed date/time edit.
- per-set `syncState` (`pending_create|synced|pending_update|pending_delete`).

**Directional heals (explicit, never automatic):** `reconcileOccurrenceList`
(local → server; re-POST local list, history-safe via the prune guard) and
`rehydrateLocalFromServer` (server → local; wipe local session, rebuild from the
server copy). We **detect and warn, never auto-resolve** — local isn't reliably
the source of truth (after a wipe it hydrates from the server).

**`healSingletonDropGroups`** runs on log-page load: nulls any orphaned singleton
drop group (legacy stray "+ Drop" tags), syncing the clear idempotently.

**IndexedDB migrations** live in `migrateSessionDb(db, oldVersion)` and are
**additive by default** — the destructive `oldVersion < 4` drop is a documented
historical one-off; new versions are `if (oldVersion < N)` create-only blocks.
Guard test: a future bump preserves data (proven in isolation).

## 7. Local vs production

Vercel auto-deploys `main`, but **migrations are manual** — shipped code can
outrun the prod schema. `/api/health` returns 503 when `applied <
EXPECTED_MIGRATIONS`; `db:check` fails on drift. **Migrate-then-deploy**: apply
the migration to prod (direct Neon endpoint) *before/at* the deploy that needs
it, else the affected API 500s. Additive columns are safe; propose before any
prod write. There is no automated prod-migration step yet — a manual runbook is
the current mechanism (see DECISIONS).

## 9. Known gaps / in-flight (hand-written — keep honest)

- **Hand-curated substitutions are documentation-only.** `exercise_substitutions`
  carries ~71 candidates but `candidate_exercise_id` is null; the live engine
  computes candidates from pattern+muscle+equipment instead. The **judgment** in
  it (back-friendly stand-ins: leg press if back flares, Pallof as the
  lumbar-safe rotation) was the only irreproducible asset and is now extracted to
  durable prose in [`SUBSTITUTION-JUDGMENT.md`](SUBSTITUTION-JUDGMENT.md) — the
  table references exercises by name (which change), so the judgment lives in
  docs, not the decaying table. Table retained as documentation-only; not wired
  into the engine (that would create a second, stale substitution source).
- **`skill_level` populated (provenance is weak).** The substitution tie-break
  reads it; it's now backfilled from free-exercise-db's `level`
  (beginner/intermediate/expert) for every library-paired exercise + a few
  obvious hand-tags (526/293/57; 4 junk/dup rows left null). The tie-break is now
  functional but the library grading is **unverified** — treat as noisy. Watch:
  if substitution ranking gets visibly worse, revisit the source (see DECISIONS).
- **Multi-device divergence — detect-and-warn (Part 3, built).**
  `hydrateFromServer` still no-ops when a local copy exists (never clobbers local
  edits), but the sessions list now *detects* the server-ahead case:
  `isDeviceBehind()` flags a session when the server has more occurrences than
  this device AND local is provably clean (finishSynced, no dirty list/meta/
  conflict). It routes to **Pull** (adopt the server) and still offers **Keep this
  device** (Reconcile, history-safe push) — the user picks the direction; never
  auto-heals. Refuses to claim "behind" on any two-sided edit (falls to the push
  path). Detection is occurrence-count level (cheap, from the list payload);
  set-level divergence within equal occurrence counts isn't surfaced here.
- **Program target chips don't survive a reinstall** — occurrence
  `targetSets/repRange/rirTarget` hydrate as null (program metadata, not logged
  input). **Closed by owner decision** — targets stay at the program level.
- **Per-unit load increment.** `defaultLoadIncrement()` keys off the `load_type`
  taxonomy (the last taxonomy in core); the Equipment model now makes increment a
  property of the *unit*. Clean end state: the adapter passes an increment from
  the equipment instance so core stops keying on `load_type`. Deferred.
- **Legacy `completed` IndexedDB store** — dead mirror; remove.
- Some historical `set_logs.equipment_type` are null on named-machine sets (type
  recoverable via the equipment row) — cosmetic; self-corrects on edit.

## 11. Traps (things that waste time or cause damage)

- **IndexedDB migrations must stay additive** — `migrateSessionDb` creates only;
  never `deleteObjectStore` a store carrying data forward (silently eats unsynced
  in-flight work). The `oldVersion < 4` drop is a historical one-off, not a
  template. Keep the data-loss guard test green.
- **`sync()` must stay serialized** (`syncChain`) or duplicate logged sets return.
- **Occurrences sync before sets** — set-logs link by `clientInstanceId`, which
  must exist server-side first.
- **The prune guard is load-bearing** — `/api/session-exercises` must never delete
  an occurrence that still has logged sets/cardio (a stale/wiped client re-POST
  would otherwise destroy history). The paired client two-pass heals legit removes.
- **`finished_at` is not a display anchor** — it re-stamps on every finish. Show/
  sort by `date` + `firstFinishedAt`. All user-facing times render **local**
  (client components + `toLocale*`); date-boundary logic uses local calendar parts
  (UTC `toISOString().slice(0,10)` files evening sessions to the next day).
- **Never mass-delete or migrate prod without sign-off.** Migrate-then-deploy.
- **Keep exercise & equipment ids stable** — logged history references them.
- **drizzle-kit `generate` needs current snapshots** — hand-writing a migration
  without regenerating the snapshot makes the next `generate` hit an interactive
  rename prompt (non-TTY = crash). If you hand-write SQL, keep the snapshot honest.
- **`_resetDbForTests()` is test-only** (deletes the whole IDB).
- **Secrets:** env-var names only in docs; the Neon credential is exposed +
  unrotated (owner's deferred task) — never commit connection strings.
